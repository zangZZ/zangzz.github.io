target-source-combina

插件式（pf4j）

结构如下
source api: 首先定义好source api（interface），打包，用于后续主工程和plugin工程引入依赖
plugins: 引入source api依赖，使用接口，在实现类上打上指定注解用于自动发现
main progress: 引入pf4j框架和source api依赖，可以热加载plugin Jar包，调用指定方法执行plugin的实现

优点：热加载jar包；完全的解耦；自由度极高；内部调用soure api；
缺点：需要手动维护数据库连接和服务bean（对于用惯了spring框架的我们来说稍显繁琐）

sdk式

结构如下
source lib：定义了source api， 引入plugin lib的实现，用于被主程序调用，可以帮助controller和具体的source 实现解耦
plugin lib：source api的实现，spring框架（类似common-lib）
main progress：引入source lib，通过spring管理source api的实现

优点：半解耦；plugin中可以使用spring管理bean；内部调用source api；
缺点：仍然需要手动维护数据库连接（因为不同的plugin可能使用不同地址的同类型数据库）；可能发生bean冲突（plugin之间可能定义重复bean）;

服务接口式（就地取材）

认为source api均来自于注册到nacos上的某个服务，在主程序配置文件中添加source type和source service的映射列表。

优点：简单（在任意服务上利用现成的代码可以很轻松的添加一个source api接口）；
缺点：系统耦合度偏高；source api调用跨服务；
